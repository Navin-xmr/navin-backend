⚠ Critical Findings 6
C-01 · Six endpoints entirely missing from Swagger CRITICAL

The following routes are implemented in route files and tested, but have zero documentation in docs/swagger.yaml. Any developer or API client relying solely on the spec cannot discover or use them.

  • POST /api/payments — create payment record
  • GET /api/payments/{id} — get single payment
  • PATCH /api/payments/{id}/status — update payment status
  • GET /api/telemetry/thresholds — fetch alert thresholds
  • POST /api/telemetry/bulk — bulk telemetry ingest
  • GET /api/shipments/{id} — get single shipment (expected by frontend)
Fix: Add all six paths to docs/swagger.yaml under paths:. The Payments endpoints should mirror the existing GET /api/payments pattern. Telemetry endpoints should document requireAuth security and their request/response schemas.
C-02 · Response envelope inconsistency — anomalies module CRITICAL

Every other endpoint returns the standard envelope { success, message, data, meta } via sendResponse(). The GET /api/anomalies Swagger spec documents a flat root response { data: [], nextCursor, hasMore }, while the frontend compatibility guide confirms the actual controller uses the sendResponse wrapper — placing nextCursor and hasMore inside meta, not the root. This causes a runtime type error on the frontend.

// Swagger says (incorrect):
{ data: Anomaly[], nextCursor: string, hasMore: boolean }

// Actual sendResponse output:
{ success: true, message: "...", data: Anomaly[], meta: { nextCursor, hasMore } }
Fix: Either update the Swagger schema to match the real sendResponse envelope and update the frontend service to read res.data.data / res.data.meta, or adjust the anomaly controller to bypass sendResponse and return the flat structure the frontend expects.
C-03 · Anomaly resolve returns non-standard key anomaly instead of data CRITICAL

PATCH /api/anomalies/{id}/resolve is documented in Swagger as returning { anomaly: Anomaly }. Every other endpoint in the system returns { data: T }. This breaks any client that destructures a consistent envelope, and forces the frontend to special-case this endpoint.

Fix: Change the Swagger response schema key from anomaly to data and ensure the controller uses sendResponse for consistency.
C-04 · Password minimum length inconsistency across auth endpoints CRITICAL

Three separate endpoints document different password minimum lengths:

  • POST /api/auth/signup — minLength: 8
  • POST /api/users — minLength: 6
  • POST /api/users/invitations/accept — minLength: 6

This means an account created via /api/users with a 6-character password cannot log in if the same minLength: 8 validation is applied at login. It also confuses developers about the actual security policy.

Fix: Standardise to one value (recommend 8 as the higher security bar) across all Zod validation schemas and all Swagger docs. Check auth.validation.ts, users.validation.ts, and update all three Swagger path entries.
C-05 · POST /api/shipments uses full Shipment schema as request body CRITICAL

POST /api/shipments has $ref: '#/components/schemas/Shipment' as its request body. The Shipment schema contains response-only fields (_id, status, milestones, offChainMetadata, createdAt, updatedAt). This misrepresents what the client should send, and the compatibility guide confirms the frontend only sends { origin, destination, enterpriseId, logisticsId, milestones?, offChainMetadata? }.

Fix: Create a separate CreateShipmentRequest component schema with only the writable fields and use that as the request body $ref. Keep the full Shipment schema for responses.
C-06 · No servers block in OpenAPI spec CRITICAL

docs/swagger.yaml has no servers: section at all. OpenAPI tools (Swagger UI, code generators, Postman import) fall back to a relative URL, making the spec unusable as a standalone reference without manual configuration. This is a structural requirement of OpenAPI 3.0.

Fix: Add a servers: block immediately after info:: 
servers:
  - url: http://localhost:3000
    description: Local development
  - url: https://api.navin.io
    description: Production
⚠ High Priority Findings 5
H-01 · Pagination strategies are inconsistent across the API HIGH

Four different pagination styles are used with no clear rule for when each applies:

  • Offset (page + limit + total): /api/shipments, /api/users
  • Cursor (cursor + hasMore + nextCursor): /api/payments, /api/anomalies
  • Both simultaneously (cursor OR page + limit): /api/telemetry — accepts both but has no documented precedence rule

The meta object structure also varies: some use { page, limit, total }, others use { nextCursor, hasMore, total }, and telemetry returns { nextCursor, hasMore, page } — a hybrid that is never fully explained.

Fix: Document a pagination strategy decision in the spec's info.description or a dedicated x-pagination-guide extension. Standardise on cursor-based for large collections and offset for small admin lists. Remove the dual-mode from telemetry or clearly document cursor takes precedence when both are supplied.
H-02 · Missing security declarations on authenticated endpoints HIGH

Multiple endpoints have requireAuth middleware in their route files but no security: block in Swagger, meaning code-generated clients will not include auth headers and security scanners will not flag them as requiring authentication:

  • GET /api/shipments — no security in Swagger
  • PATCH /api/shipments/{id} — no security in Swagger
  • GET /api/telemetry — no security in Swagger (route uses requireAuth)
  • GET /api/telemetry/thresholds — missing entirely
  • POST /api/telemetry/bulk — missing entirely
Fix: Add security: [{ bearerAuth: [] }] to every protected path operation, or add a global security: block at the top level and explicitly opt out (using security: []) only on public endpoints (/api/health, /api/auth/signup, /api/auth/login).
H-03 · RBAC roles not documented on restricted endpoints HIGH

Many endpoints use requireRole() middleware but the required roles are either missing from descriptions or only buried in plain-text descriptions with no machine-readable format:

  • POST /api/auth/api-keys — ADMIN/SUPER_ADMIN required, not in spec
  • POST /api/payments — missing from spec entirely
  • PATCH /api/payments/{id}/status — missing from spec entirely
  • POST /api/users/team — description says ADMIN/SUPER_ADMIN but no structured 403 response schema
  • GET /api/users — description mentions ADMIN/MANAGER/SUPER_ADMIN but no 403 response
Fix: Add a 403 response to all role-restricted operations. Use OpenAPI extensions like x-required-role: [ADMIN, SUPER_ADMIN] or document required roles in a structured description table to make RBAC requirements machine-readable.
H-04 · Webhook IoT response breaks the standard envelope HIGH

POST /api/webhooks/iot 201 response is documented as { data: { _id, shipmentId, … } } — missing the success and message fields that every other endpoint includes. This makes the response un-parseable by generic envelope-aware frontend code.

Fix: Update the Swagger 201 schema to include success: boolean and message: string at the root level, matching the sendResponse pattern used by every other module.
H-05 · info block is critically sparse HIGH

The top-level info section contains only title and version. Missing fields that all OpenAPI 3.0 consumers expect:

  • description — no overview, architecture notes, or authentication guide
  • contact — no email or support URL
  • license — no license information

The bearerAuth security scheme also lacks a bearerFormat: JWT hint, which Swagger UI uses to populate the Authorize dialog correctly.

Fix: Expand the info block with a description that includes base URL, auth flow summary, and pagination conventions. Add bearerFormat: JWT to the bearerAuth scheme.
⚠ Medium Priority Findings 5
M-01 · Multiple invitation endpoint responses have no body schema MEDIUM

Three invitation-flow endpoints document only an HTTP status code with no response body:

  • POST /api/users/invitations → "201 Invitation generated" — no schema
  • GET /api/users/invitations/verify → "200 Invitation token is valid" — no schema
  • POST /api/users/invitations/accept → "201 Invitation accepted" — no schema

Developers have no way to know what the server returns on success.

Fix: Add response body schemas using $ref: '#/components/schemas/ErrorResponse' for errors and inline/component schemas for success responses. At minimum document the { success, message, data } envelope shape.
M-02 · Redundant / overlapping user creation endpoints not differentiated MEDIUM

Three endpoints create users but their distinction is unclear in the docs:

  • POST /api/auth/signup — described as "Register a new user"
  • POST /api/users — described as "Register a new user account. Public endpoint for self-signup or admin-created users"
  • POST /api/users/team — described as "Create a user within the organization"

The first two descriptions are nearly identical. There is no guidance on which one the frontend or an IoT client should use, creating confusion at integration time.

Fix: Clearly differentiate in descriptions: /auth/signup is for unauthenticated self-registration; /users is admin-created without org context; /users/team is admin-created within a specific org. Consider deprecating the duplicate if they serve the same purpose.
M-03 · GET /api/shipments missing authentication mention; frontend-backend pagination mismatch MEDIUM

The compatibility guide explicitly flags that the frontend's shipmentApi.getAll expects offset pagination returning { data: Shipment[], page, limit, total } at the res.data.data level. The backend returns data at res.data.data and meta at res.data.meta — a two-level nesting the frontend doesn't expect.

Additionally, Swagger has no security block for this path despite the route being authentication-gated in production.

M-04 · Internal documentation conflicts between Swagger and compatibility guide MEDIUM

backend-compatibility-guide-check.md (section 8.6) states: "The backend has a users module but only defines POST /api/users and DELETE /api/users/:id, with no GET /api/users route." This is factually wrong — GET /api/users exists in both the Swagger spec and users.routes.ts. When internal docs contradict each other, developers lose trust in all documentation.

Fix: Audit and correct backend-compatibility-guide-check.md. Keep a single source of truth (Swagger) and auto-generate compatibility notes rather than maintaining them manually.
M-05 · Error response coverage is inconsistent across endpoints MEDIUM

Error response documentation varies significantly. Some endpoints document 400/401/403/404 fully; others document only one or two. Examples of gaps:

  • POST /api/shipments — documents only 201; no 400/401/403
  • PATCH /api/shipments/{id}/status — documents 200 and 400 only; no 401/403
  • GET /api/users/invitations/verify — 200 and 401 only; no 400 for malformed token
  • GET /api/shipments/{id} — entirely missing
Fix: Establish a baseline set of error responses for each endpoint category: public endpoints need 400; authenticated endpoints need 400/401; role-restricted endpoints need 400/401/403; resource endpoints need 400/401/403/404.

Bottleneck Analysis
B-01 · Telemetry GET supports dual pagination modes with no documented precedence HIGH

GET /api/telemetry accepts both cursor (cursor-based) and page (offset-based) simultaneously. The Swagger spec documents both query parameters but gives no guidance on what happens when both are supplied, which takes precedence, or when a developer should choose one over the other. The response meta object also returns both page and nextCursor simultaneously, creating an ambiguous response shape.

This is a developer experience bottleneck: integrators must reverse-engineer the priority rule from source code rather than the spec.

B-02 · No global rate-limit documentation except login MEDIUM

POST /api/auth/login documents a 429 response with a retryAfter field. No other endpoint documents rate limits, even though the IoT webhook endpoint is a high-frequency ingest path that almost certainly has (or needs) rate limiting. Clients building IoT integrations have no guidance on safe request cadence or backoff strategy.

B-03 · Stellar anchoring latency not documented for telemetry consumers MEDIUM

The telemetry response includes anchorStatus: "PENDING_ANCHOR" | "ANCHORED" | "ANCHOR_FAILED" and stellarTxHash, indicating asynchronous Stellar anchoring. However, there is no documentation of: the expected anchoring latency, whether clients should poll, how to correlate a PENDING_ANCHOR record with its final ANCHORED state, or whether the WebSocket telemetry_update event fires again when anchoring completes. This is a functional bottleneck for clients that need to display on-chain confirmation.

docs/websockets.md provides full TypeScript interfaces for every event payload, connection handshake examples, room join/leave patterns, error codes, and cleanup behaviour. This is the highest-quality documentation in the project.

S-02 · ErrorResponse component schema is well-defined and reused correctly PASS

The components/schemas/ErrorResponse schema is correctly defined with required fields, used via $ref in most error responses, and has example values. This is the right approach and should be extended to cover success envelopes.

S-03 · Anomaly and payment filter parameters are well documented PASS

GET /api/anomalies and GET /api/payments document all query parameters with types, enums, defaults, and descriptions. This level of detail significantly reduces integration guesswork.

S-04 · Security schemes are correctly defined and structured PASS

All three security schemes (bearerAuth, apiKeyAuth, stellarSignature) are correctly declared in components/securitySchemes with header names, types, and descriptions. The separation of IoT device auth from user JWT auth is architecturally correct.

S-05 · X-Request-ID header documented in components PASS

The X-Request-ID tracking header is documented in components/headers with a clear description and example. It is correctly referenced in the GET /api/shipments response headers. Extend this reference to all endpoint responses for full consistency.
